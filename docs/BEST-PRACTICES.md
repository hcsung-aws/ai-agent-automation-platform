# Best Practices: 실패 사례와 교훈

Agent 시스템을 구축하면서 겪은 실패 사례와 교훈을 정리합니다.

---

## 1. Agent 설계 실패

### 실패: 너무 많은 기능을 한 Agent에

```
❌ "만능 Agent"
- 휴가 조회 + 서버 모니터링 + 데이터 분석 + 리포트 생성
- 도구 20개, 프롬프트 2000자
```

**문제**: 
- Agent가 어떤 도구를 써야 할지 혼란
- 응답 품질 저하
- 디버깅 어려움

**교훈**:
```
✅ 영역별로 Agent 분리
- HR Agent: 휴가, 직원 정보
- DevOps Agent: 서버, 알람
- 각 Agent는 도구 5-10개 이내
```

---

### 실패: 모호한 도구 설명

```python
❌
@tool
def get_data(id: str) -> str:
    """데이터를 가져옵니다."""  # 무슨 데이터?
```

**문제**: Agent가 언제 이 도구를 써야 할지 모름

**교훈**:
```python
✅
@tool
def get_leave_balance(employee_id: str) -> str:
    """직원의 휴가 잔여일을 조회합니다.
    
    Args:
        employee_id: 직원 ID (예: E001, E002)
    
    Returns:
        휴가 잔여일 정보 (이름, 잔여일수)
    """
```

---

## 2. Supervisor 실패

### 실패: 위임 규칙 없음

```
❌ "적절한 Agent에게 위임하세요"
```

**문제**: 
- "서버 담당자 휴가 확인" → HR? DevOps?
- 일관성 없는 위임

**교훈**:
```
✅ 명확한 키워드 기반 규칙

### HR Agent
키워드: 휴가, 직원, 급여, 인사
예시: "휴가 잔여일", "직원 정보"

### DevOps Agent  
키워드: 서버, 알람, EC2, 배포
예시: "서버 상태", "알람 목록"
```

---

### 실패: Supervisor가 응답을 과도하게 요약

```
❌ 
사용자: "알람 목록 보여줘"
DevOps Agent: "CRITICAL: DB-Connection (3건), WARNING: CPU-High (2건)..."
Supervisor: "알람이 있습니다."  # 정보 손실!
```

**문제**: 중요한 세부 정보가 사라짐

**교훈**:
```
✅ 상세 보기 버튼 추가
- Supervisor 요약은 유지
- "상세 보기" 버튼으로 전체 응답 확인 가능
```

---

## 3. 피드백 루프 실패

### 실패: 피드백 수집만 하고 분석 안 함

```
❌ 
- 피드백 100건 축적
- 아무도 안 봄
- Agent 개선 없음
```

**문제**: 피드백이 있어도 개선으로 이어지지 않음

**교훈**:
```
✅ 주간 피드백 리뷰 루틴
1. 매주 월요일 부정 피드백 확인
2. 3회 이상 반복 패턴 파악
3. Agent Builder로 개선 적용
4. 다음 주 효과 확인
```

---

### 실패: 피드백 없이 추측으로 개선

```
❌
"사용자가 이런 기능을 원할 것 같아서 추가했어요"
→ 아무도 안 씀
```

**문제**: 실제 니즈와 다른 기능 개발

**교훈**:
```
✅ 피드백 기반 개선만
- 부정 피드백에서 패턴 발견 → 개선
- 긍정 피드백 많은 기능 → 유지/강화
- 추측 기반 기능 → 보류
```

---

## 4. 자연어 Agent 생성 실패

### 실패: 한 번에 완벽한 Agent 요청

```
❌
"HR Agent 만들어줘. 휴가 조회, 급여 명세, 채용 현황, 
 인사 평가, 교육 이력, 복리후생 안내, 조직도 조회..."
```

**문제**: 
- Agent Builder가 혼란
- 생성된 코드 품질 저하
- 테스트 어려움

**교훈**:
```
✅ 단계별 확장
1단계: "휴가 조회만 있는 HR Agent 만들어줘"
2단계: (테스트 후) "급여 명세 기능 추가해줘"
3단계: (테스트 후) "채용 현황 기능 추가해줘"
```

---

### 실패: 맥락 없이 수정 요청

```
❌
"Agent 고쳐줘"  # 무슨 Agent? 뭘 고쳐?
```

**문제**: Agent Builder가 무엇을 수정해야 할지 모름

**교훈**:
```
✅ 구체적인 맥락 제공
"HR Agent의 휴가 조회 기능에서 
 직원 ID가 없을 때 에러가 나. 
 ID 없으면 목록을 보여주도록 수정해줘"
```

---

## 5. 운영 실패

### 실패: 테스트 없이 배포

```
❌
Agent Builder가 코드 생성 → 바로 배포 → 장애
```

**문제**: 생성된 코드에 버그가 있을 수 있음

**교훈**:
```
✅ 테스트 → 배포 사이클
1. Agent Builder가 코드 생성
2. 로컬에서 테스트 (정상 케이스 + 예외 케이스)
3. 문제 있으면 수정 요청
4. 테스트 통과 후 배포
```

---

### 실패: 로그 없이 운영

```
❌
사용자: "Agent가 이상해요"
운영자: "뭐가 이상한데요?"
사용자: "그냥 이상해요"
```

**문제**: 문제 재현/분석 불가

**교훈**:
```
✅ 실행 로그 필수
- 모든 요청/응답 DynamoDB에 저장
- session_id로 대화 흐름 추적
- 에러 발생 시 로그로 원인 분석
```

## 체크리스트

### Agent 설계
- [ ] 영역별로 Agent 분리 (도구 5-10개 이내)
- [ ] 도구 설명 상세히 작성
- [ ] 예시 입력/출력 포함

### Supervisor
- [ ] 키워드 기반 위임 규칙 명시
- [ ] 복합 요청 처리 방법 정의
- [ ] 상세 보기 옵션 제공

### 피드백
- [ ] 피드백 수집 체계 구축
- [ ] 주간 리뷰 루틴 설정
- [ ] 피드백 기반 개선만 진행

### 자연어 생성
- [ ] 단계별로 기능 추가
- [ ] 구체적인 맥락 제공
- [ ] 테스트 후 다음 단계 진행

### 운영
- [ ] 로컬 테스트 필수
- [ ] 실행 로그 저장
- [ ] 에러 모니터링

---

## 핵심 원칙

```
1. 작게 시작하고 점진적으로 확장
2. 피드백 없이 추측하지 않기
3. 테스트 없이 배포하지 않기
4. 로그 없이 운영하지 않기
5. 한 번에 완벽하려 하지 않기
```
